# MaRsiComboStrategy 设计文档

## 1. 策略概述

**MA-RSI组合策略** 是专门针对上期所黄金期货设计的技术分析策略，结合双均线趋势识别和RSI超买超卖指标，提供稳健的交易信号。

### 1.1 适用市场
- ✅ 上期所黄金期货（au主力合约）
- ✅ 日内交易和短线交易
- ✅ 趋势性行情和震荡行情
- ⚠️ 需要足够的流动性和波动性

### 1.2 风险特征
- 📊 **风险等级**：中等
- 💰 **资金要求**：适中（每手约5-10万保证金）
- ⏱️ **持仓周期**：分钟级到小时级
- 📈 **预期收益**：稳健型，追求风险调整后收益

---

## 2. 核心交易逻辑

### 2.1 信号生成流程

```
K线数据 → 更新ArrayManager → 计算技术指标 → 检测金叉/死叉 → RSI确认 → 生成交易信号
```

### 2.2 技术指标组合

| 指标 | 参数 | 用途 |
|------|------|------|
| EMA5 | 5周期 | 短期趋势，捕捉快速变化 |
| EMA20 | 20周期 | 长期趋势，确认主要方向 |
| RSI14 | 14周期 | 超买超卖确认，避免极端点位入场 |

### 2.3 信号生成条件

| 信号类型 | 条件 |
|----------|------|
| 🟢 **买入信号** | 金叉确认（EMA5上穿EMA20，2根K线确认）+ RSI在30-70区间 |
| 🔴 **卖出信号** | 死叉确认（EMA5下穿EMA20，2根K线确认）+ RSI在30-70区间 |
| ⏸️ **观望信号** | 无交叉信号 或 RSI处于极端区域（<30 或 >70） |

## 3. 交叉检测机制

### 3.1 问题背景

金叉死叉检测面临的核心矛盾：
- **阈值高** → 过滤假突破，但错过正常行情（黄金价格波动缓慢）
- **阈值低** → 捕捉更多机会，但假突破增多

### 3.2 解决方案：两阶段时间确认机制

**2025-12-30 重大改进（含极值确认优化）**

```
┌──────────────────────────────────────────────────────────────┐
│  第一阶段：检测交叉                                            │
│  ────────────────────────                                     │
│  发现 MA5 穿越 MA20（穿越幅度 > 0.01%）                         │
│        ↓                                                      │
│  进入"待确认"状态，记录：                                       │
│  • 初始差值（MA5-MA20）                                        │
│  • 交叉时收盘价                                                │
│  • 初始极值（金叉=最高价，死叉=最低价）                          │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  第二阶段：等待2根K线确认                                       │
│  ────────────────────────────                                 │
│  每根K线更新极值：                                              │
│  • 金叉：更新期间最高价（取max）                                 │
│  • 死叉：更新期间最低价（取min）                                 │
│                                                               │
│  检查1：MA5是否仍在正确一侧？                                    │
│    - 否 → ❌ 假突破，取消信号                                   │
│    - 是 → 继续检查                                             │
│                                                               │
│  检查2（达到确认K线数时）：                                      │
│  • 差值是否保持（≥初始值50%）？                                  │
│  • 极值确认：                                                   │
│    - 金叉：期间最高价 > 交叉时收盘价？                           │
│    - 死叉：期间最低价 < 交叉时收盘价？                           │
│    - 都满足 → ✅ 确认交叉，发出信号                              │
│    - 否则 → ❌ 信号不可靠，取消                                 │
└──────────────────────────────────────────────────────────────┘
```

**极值确认的优势**：
- 不要求每根K线收盘价都比交叉价更高/更低
- 只要确认期间曾触及更高/更低点，就说明市场确实在试探这个方向
- 允许短期波动，不会因小幅反弹/回调而错过真正的趋势

### 3.3 交叉幅度计算

交叉幅度 = 从交叉前到交叉后的**总穿越距离**：

```python
diff_before = MA5[前1时刻] - MA20[前1时刻]  # 交叉前差值
diff_after = MA5[当前时刻] - MA20[当前时刻]  # 交叉后差值
cross_distance = abs(diff_after - diff_before) / MA20[当前时刻]
```

**示例**：
```
交叉前：MA5 = 974.88, MA20 = 975.28, 差值 = -0.40（在下方）
交叉后：MA5 = 975.53, MA20 = 975.43, 差值 = +0.10（在上方）
总穿越距离 = |0.10 - (-0.40)| = 0.50元
cross_distance = 0.50 / 975.43 = 0.0512%
```

### 3.4 相关参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `cross_confirm_bars` | 2 | 确认所需K线数 |
| `min_cross_distance` | 0.0001 (0.01%) | 最小穿越幅度阈值（极低，主要靠时间确认过滤假突破） |

## 4. 持仓管理逻辑

### 4.1 信号处理完整流程

**金叉信号（BUY）处理流程**：
```
BUY信号
│
├── 1. 查询持仓信息（获取真实均价）
│
├── 2. 检查空头持仓（反向持仓处理）
│   ├── 有空头且盈利 → ✅ 平仓止盈
│   └── 有空头但亏损 → 🔒 保留，等待ATR加仓
│
└── 3. 检查是否开多（同向持仓处理）
    ├── 多头达上限(max_position) → ⚠️ 不开仓
    ├── 无多头持仓 → ✅ 直接开仓
    ├── 有多头且亏损 >= ATR阈值 → ✅ 加仓摊低成本
    └── 有多头但亏损 < ATR阈值 → ⚠️ 不加仓
```

**死叉信号（SELL）处理流程**：
```
SELL信号
│
├── 1. 查询持仓信息（获取真实均价）
│
├── 2. 检查多头持仓（反向持仓处理）
│   ├── 有多头且盈利 → ✅ 平仓止盈
│   └── 有多头但亏损 → 🔒 保留，等待ATR加仓
│
└── 3. 检查是否开空（同向持仓处理）
    ├── 空头达上限(max_position) → ⚠️ 不开仓
    ├── 无空头持仓 → ✅ 直接开仓
    ├── 有空头且亏损 >= ATR阈值 → ✅ 加仓摊低成本
    └── 有空头但亏损 < ATR阈值 → ⚠️ 不加仓
```

### 4.2 盈亏计算公式

| 持仓方向 | 盈亏计算 | 说明 |
|---------|---------|------|
| 多头 | `(当前价 - 均价) × 手数` | 正数=盈利，负数=亏损 |
| 空头 | `(均价 - 当前价) × 手数` | 正数=盈利，负数=亏损 |

**示例**：
```
多头持仓：均价1000元，当前价1005元，持仓1手
盈亏 = (1005 - 1000) × 1 = +5元（盈利）

空头持仓：均价1000元，当前价1005元，持仓1手
盈亏 = (1000 - 1005) × 1 = -5元（亏损）
```

### 4.3 平仓/保留决策逻辑

**核心策略**：反向信号只平盈利仓位，亏损仓位保留

| 场景 | 处理方式 | 原因 |
|------|---------|------|
| 反向信号 + 盈利 | 平仓止盈 | 锁定利润 |
| 反向信号 + 亏损 | 🔒 保留持仓 | 等待同方向信号加仓摊低成本 |
| 同方向信号 + 亏损达ATR | 加仓 | 摊低均价 |

**设计思路**：马丁格尔 + 对冲策略
```
持有多头亏损
├── 出现金叉 + 亏损 >= ATR×1.5 → 加仓摊低成本
├── 出现死叉 → 不平多（亏损）→ 开空对冲
└── 价格回升 → 等待盈利平仓
```

### 4.4 对锁说明

**允许同时持有多空仓位**（对锁）

当出现以下情况时会形成对锁：
1. 持有多头亏损 → 出现死叉 → 不平多 → 开空
2. 持有空头亏损 → 出现金叉 → 不平空 → 开多

**对锁的作用**：
- 用反向仓位对冲风险
- 等待行情明朗后择机解锁
- 避免在亏损时强制平仓造成实际损失

### 4.5 加仓控制机制

为避免在震荡行情中频繁加仓被套，策略只在以下情况允许加仓：
- ✅ 必须有新的同方向交叉信号（金叉/死叉）
- ✅ 无同方向持仓 → 直接开仓
- ✅ 已有持仓且亏损达到 **ATR动态阈值** → 加仓

**2025-01-03 改进：ATR动态阈值**

旧方案：固定百分比（0.2%），在震荡行情中太小，容易被套

新方案：基于 ATR 的动态阈值
```python
# 多头加仓条件
price_loss = long_price - current_price  # 亏损金额（正数=亏损）
atr_threshold = ATR(14) × add_loss_atr_multiplier
should_add = price_loss >= atr_threshold

# 空头加仓条件
price_loss = current_price - short_price  # 亏损金额（正数=亏损）
atr_threshold = ATR(14) × add_loss_atr_multiplier
should_add = price_loss >= atr_threshold
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `atr_period` | 14 | ATR计算周期 |
| `add_loss_atr_multiplier` | 1.5 | 阈值倍数 |

**优点**：
- 波动大时阈值高，避免频繁加仓
- 波动小时阈值低，不错过加仓机会
- 自适应市场，比固定百分比更合理

### 4.6 真实均价管理

**2026-01-06 改进**：自行记录真实开仓均价

**问题**：CTP返回的 `position.price` 是**结算价**，不是真实开仓均价
- 每天下午收盘后，CTP会将持仓价格重置为结算价
- 这导致盈亏计算、加仓判断、止盈止损都不准确

**解决方案**：策略内部维护真实均价
```python
# 数据结构
self.real_positions = {
    "long": {"volume": 2, "avg_price": 1000.5, "total_cost": 2001.0},
    "short": {"volume": 1, "avg_price": 1005.0, "total_cost": 1005.0}
}

# 持久化存储
文件路径: data/real_positions_{strategy_name}_{symbol}.json
```

**更新时机**：
- 开仓成交 → 加权平均计算新均价
- 平仓成交 → 减少持仓量（均价不变）
- 全平 → 删除记录
- 服务重启 → 从JSON文件恢复

**均价计算公式**（加仓时）：
```
新均价 = (原持仓量 × 原均价 + 新成交量 × 成交价) / (原持仓量 + 新成交量)
```

### 4.7 智能平仓（上期所特殊处理）

上期所区分今仓/昨仓，智能平仓自动拆分订单：
```
平仓请求 → 查询今昨仓 → 优先平昨仓 → 再平今仓
```

**拆分逻辑**：
```python
if today_pos > 0 and yesterday_pos > 0 and volume > yesterday_pos:
    # 需要拆分：先平昨仓，再平今仓
    平昨仓(min(volume, yesterday_pos))
    time.sleep(0.1)  # 避免订单冲突
    平今仓(volume - yesterday_volume)
else:
    # 不需要拆分
    直接平仓(volume)
```

### 7.1 信号间隔边界条件 (2025-12-29)

**问题**：使用 `<` 导致恰好在间隔时间发出信号
```python
# 修复前
if time_since_last_signal < self.min_signal_interval:
    
# 修复后
if time_since_last_signal <= self.min_signal_interval:
```

### 7.2 重复开仓问题 (2025-12-29)

**问题**：在已有同方向持仓时继续发送开仓信号

**修复**：在 `_process_trading_signal` 开头增加检查
```python
if action == 'BUY' and long_position >= self.max_position:
    return  # 跳过信号
if action == 'SELL' and short_position >= self.max_position:
    return  # 跳过信号
```

### 7.3 交叉幅度计算错误 (2025-12-30)

**问题**：只计算交叉后的距离，而非总穿越距离
```python
# 修复前（错误）
cross_distance = abs(ma5_values[1] - ma20_values[1])  # 只有0.10

# 修复后（正确）
diff_before = ma5_values[0] - ma20_values[0]
diff_after = ma5_values[1] - ma20_values[1]
cross_distance = abs(diff_after - diff_before)  # 0.50
```

### 7.4 假突破过滤不足 (2025-12-30)

**问题**：交叉幅度阈值太高会错过正常行情，太低会有假突破

**修复**：实现两阶段时间确认机制
- 降低幅度阈值到0.01%
- 增加2根K线确认期
- 检查差值保持和价格确认

### 7.5 MA历史更新时机错误 (2025-12-30)

**问题**：
1. MA历史在 `_analyze_market_conditions` 中更新，但该函数只在信号间隔通过后才调用
2. `_detect_ma_cross` 在 `on_bar_impl` 中就被调用，此时MA历史还没更新
3. 如果信号间隔通过，同一K线内 `_detect_ma_cross` 被调用两次，导致 `pending_cross_bar_count` 增加两次

**修复**：
1. 把MA历史更新移到 `on_bar_impl`，在调用 `_detect_ma_cross` 之前
2. `_detect_ma_cross` 只调用一次，结果缓存到 `current_cross_signal`
3. `_analyze_market_conditions` 使用缓存的结果，不再重复调用

```python
# on_bar_impl 中的正确顺序
self.ma5_history.append(self.current_ma5)     # 先更新历史
self.ma20_history.append(self.current_ma20)
self.current_cross_signal = self._detect_ma_cross(bar.close_price)  # 再检测交叉
```
